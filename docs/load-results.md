# Отчёт по результатам нагрузочного тестирования

## Информация

- Грубая линейная эвристическая оценка зависимости пиковой частоты запросов сервису
  от количества реальных пользователей: 10 RPS - 600 пользователей, 30 RPS - 1800
  пользователей, 100 RPS - 6000 пользователей. (1)
- Тесты производились на Docker симуляции VPS: VPS_1 (1 ядро, 2 ГБ ОЗУ),
  VPS_4 (4 ядра, 8 ГБ), VPS_8 (8 ядер, 16 ГБ) — для сценария `/stats` на высокой
  частоте запросов.
- Docker контейнер симуляции VPS, помимо приложения friends_bot_service,
  содержит базу данных PostgreSQL и веб-сервер Nginx. Большее число ядер в
  лимите контейнера снижает конкуренцию между этими сервисами, а не только даёт
  возможность использовать воркеры.
- VPS_8 занимает все ядра хост-машины. Рядом работают k6, Docker и служебные
  процессы, поэтому абсолютные потолки RPS на VPS_8 занижены относительно
  выделенного облачного сервера с теми же 8 ядрами; сравнение A и B на одном
  стенде при этом остаётся корректным.
- Ввод воркеров во всех сценариях приводит к дополнительному обращению в БД до
  обработчика и динамическому созданию объекта Bot на каждый запрос. (2)
- Ввод воркеров в сценарии /run приводит к дополнительному обращению в БД в
  обработчике. (3)
- Ветка feature/workers предусматривает, что подключение каждого дополнительного
  воркера в работу сопровождается автоматическим увеличением количества доступных
  подключений к БД. (4)
- Таймаут пула БД: обработчик не получил свободное соединение из пула SQLAlchemy
  за 30 с (`pool_timeout`) - все слоты (`DB_POOL_SIZE` + `DB_MAX_OVERFLOW`) заняты.
  SQLAlchemy бросает `TimeoutError` (QueuePool); `run_with_unit_of_work` оборачивает
  его в `DatabaseUnavailableError` и пишет `friends_bot_db_errors_total`. (5)
- Таймаут aiogram: при обработке `/run` очередной запрос к Telegram API Mock не
  завершился за 60 с - клиент aiogram не дождался ответа из‑за перегрузки процесса
  и очереди исходящих HTTP. Выбрасывается `TelegramNetworkError` и пишется метрика
  `friends_bot_handler_invocations_total{outcome="error"}` (6)
- Количетво запросов к сервису в тестах веток main и воркеров может не совпадать.
  Grafana k6 ждёт завершения HTTP-запроса VU (virtual user) перед следующим. Если
  система у предела, фактическая частота и общее число запросов k6 падают. (7)
- Все тесты поллинга производились при 40 активных ботах. Это создаёт постоянную
  базовую нагрузку на процессор в 80%. При другом количестве активных ботов,
  результаты тестов будут иные. (8)
- Ошибки БД и Telegram считаются разными метриками. Если после таймаута пула БД
  хендлер не успевает отправить пользователю сообщение об ошибке, один запрос
  попадает в обе корзины - сумма долей обработанных и необработанных запросов
  может быть немного выше 100%. (9)
- Счётчики в Grafana снимаются через `increase(...[$__range])`. У increase бывают
  артефакты (краткий «провал» перед ростом, небольшая погрешность экстраполяции).
  Абсолютные расхождения небольшие; внутри одного прогона они обычно одинаково
  сдвигают серии. Для выводов AB этого достаточно; без increase (разность сырого
  счётчика) было бы чуть точнее, но пересъём не делался.

## Обозначения

- Git ветки для тестов: **A** - ветка main, **B(n)** - ветка feature/workers,
  n - количество воркеров.
- Docker контейнеры симуляции сервера: **VPS_1** - 1 ядро и 2 ГБ ОЗУ,
  **VPS_4** - 4 ядра и 8 ГБ ОЗУ, **VPS_8** - 8 ядер и 16 ГБ ОЗУ.

## Метрики

- **Задержка выполнения запросов** - время, прошедшее от получения обработчиком
бота запроса от k6, до его полного выполнения. Является медианой от временного
ряда с 0.95 квантилями затраченного времени. Квантили берутся через PromQL/Grafana
запрос. Медиана берётся средствами Grafana, как `Legend - Values - 50th %`.
Для webhook режима не учитывается время, затраченное HTTP API на передачу запроса
в обработчик через `feed_update`. Метрика учитывает также время до падения обработки
по таймаутам (см.5,6). Сценарий `/stats`: базовое время без нагрузки 10 мс.
Сценарий `/run`: базовое время без нагрузки 11 секунд. Всё время более 60 секунд
попадает в одну корзину: `60+ сек`.

  PromQL/Grafana запрос для Ramp Stats:

  ```promql
  histogram_quantile(0.95, rate(friends_bot_handler_duration_seconds_bucket[$__rate_interval]))
  ```

  PromQL/Grafana запрос для Ramp Run:

  ```promql
  histogram_quantile(0.95, rate(friends_bot_draw_handler_duration_seconds_bucket[$__rate_interval]))
  ```

- **Количество запросов** - накопленная сумма запросов k6, которые были переданы
обработчикам. Берётся через PromQL/Grafana запрос. В Grafana снимается максимум
`Legend - Values - Max`.

  PromQL/Grafana запрос:

  ```promql
  sum(increase(friends_bot_handler_invocations_total[$__range]))
  ```

- **Частота запросов** - количество запросов в секунду, которые были переданы от
k6 обработчикам. Берётся через PromQL/Grafana запрос. В Grafana снимаются максимум,
форма максимума (`острый пик`, `плоское плато`) и медиана.

  PromQL/Grafana запрос:

  ```promql
  sum(rate(friends_bot_handler_invocations_total[$__rate_interval]))
  ```

- **Количество выполненных запросов** - накопленная сумма запросов k6, которые
были переданы обработчикам и: для `/stats` - хендлер завершился без исключений;
для `/run` - розыгрыш был одобрен и доведён до конца (`draw_completed`). Берётся
через PromQL/Grafana запрос. В Grafana снимается максимум.

  PromQL/Grafana запрос для Ramp Stats (Текущая реализация подсчета метрики
  `friends_bot_handler_invocations_total{outcome="ok"}` включает в себя мягкое
  завершение обработчика после ошибок БД. До момента большей деталиции метрик
  требуется вычесть ошибки БД):

  ```promql
  sum(increase(friends_bot_handler_invocations_total{outcome="ok"}[$__range]))
  -
  sum(increase(friends_bot_db_errors_total[$__range]) or vector(0))
  ```

  PromQL/Grafana запрос для Ramp Run:

  ```promql
  increase(friends_bot_draw_completed_total[$__range])
  ```

- **Количество не выполненных запросов по таймауту aiogram** - накопленная сумма
запросов k6, которые были переданы обработчикам и не выполнены по причине
`TelegramNetworkError` (см.6). Берётся через PromQL/Grafana запрос. В Grafana
снимается максимум.

  PromQL/Grafana запрос:

  ```promql
  increase(friends_bot_handler_invocations_total{outcome="error"}[$__range])
  ```

- **Количество не выполненных запросов по таймауту пула БД** - накопленная сумма
запросов k6, которые были переданы обработчикам и не выполнены по причине
`DatabaseUnavailableError` (см.5). Берётся через PromQL/Grafana запрос. В Grafana
снимается максимум.

  PromQL/Grafana запрос:

  ```promql
  increase(friends_bot_db_errors_total[$__range])
  ```

### Сценарий Ramp Run

#### Параметры прогона

```.env.load
BOT_MODE=webhook
NGINX_ENABLED=1
WORKER_COUNT=1  # Или 4
VPS_SIM_CPUS=1  # Или 4
VPS_SIM_MEMORY=2g  # Или 8g
LOAD_BOT_COUNT=40
LOAD_CHATS_PER_BOT=500  # Подбирается так, чтобы уникальных сочетаний чат-бот
# было достаточно для каждого отдельного запроса k6 в рамках тестового прогона
# Например 500 чатов * 40 ботов = 20000 потенциальных розыгрышей /run

LOAD_PLAYERS_PER_CHAT=2
DB_POOL_SIZE=5
DB_MAX_OVERFLOW=10
```

```.env.k6

LOAD_RAMP_RPS_PEAK=30  # Последовательно 30, 50, 100, 200
# Время прогона и RPS вместе дают общее количетво запросов k6. Количество
# уникальных сочетаний чат-бот подбирается исходя из эти параметров
LOAD_RAMP_STAGE_RAMP_UP=30s
LOAD_RAMP_STAGE_PLATEAU=1m
LOAD_RAMP_STAGE_RAMP_DOWN=30s
LOAD_RAMP_MAX_VUS=100  # Или 200, если в процессе прогона теста была нехватка
LOAD_RAMP_PREALLOCATED_VUS=100  # Или 200
```

## Наблюдения

- Ввод воркеров снижает базовую постоянную нагрузку на ядро процессора с ~60% до
  почти 0% за счёт перевода мастер-бота с поллинга на вебхук.
- Ввод воркеров в вебхук-режиме увеличивает потребление RAM. Предположительно
  память уходит на создание экземпляров Bot на каждый запрос и удержание их до
  конца обработки. Для `/stats` рост порядка ~30%; для `/run` из‑за длинной
  обработки — более чем на 100%.
- Ветка main в вебхук сценарии `/stats` при росте целевого RPS имеет фактический
  потолок частоты запросов ~230–260 RPS, который больше не увеличивается с повышением
  целевой частоты и количества ядер процессора. При этом задержки остаются низкими
  (~100–200 мс), успех 100%. Очередь хендлера не раздувается, а лишний трафик
  просто не входит в обработку.
- Ветка воркеров в вебхук сценарии `/stats` иначе: при высоких частотах
  (400 / 500 RPS) и переходе на большее количество ядер (VPS_4 / VPS_8) у воркеров
  падает задержка и увеличивается фактическая частота RPS. Однако в целом, это всё-равно
  скорее регрессия по сравнению с main, так как задержки отностельно него выше.
  Улучшение прохода плохо масштабируется в данной архитектуре параллельности для
  `/stats`.
- В вебхук режиме наблюдается преимущество от ввода воркеров при нагрузке
  сценарием /run на сервис от 100 RPS и выше, что соответствует примерно 6000
  активным пользователям сервиса (см.1).
- Реальный трафик — смесь команд; прогоны `/stats` и `/run` — граничные случаи.
- Поллинг не тянет удовлетворительную работу при более чем 40 активных ботах (см.8)
  даже на низкой нагрузке: каждый бот держит свой long-poll и съедает CPU.

## Выводы

- **AB-1, webhook `/run`.** Выигрыш воркеров зависит от железа. На VPS_1 картина
  смешанная (на ~50 RPS воркеры часто хуже; на высокой нагрузке лучше процент
  доведённых розыгрышей, но задержки времени всё равно большие). На VPS_4 с
  `WORKER_COUNT=4` с ~100 RPS (~6000 пользователей, см.1) воркеры заметно
  выигрывают по доле успешных `/run` и устойчивости. Run - это тяжёлый сценарий,
  где обработка растянута на 11+ секунд, с большим количеством асинхронных переключений
  между корутинами внутри. Это приводит к тому что в обработку попадает большое
  количество одновременных запросов и один event loop с полной цепочкой обработки
  не справляется. Несколько параллельных процессов же имеют свои собственные
  event loop, что окупает часть накладных расходов от ввода воркеров (см.2,3).
- **AB-1, webhook `/stats`.** На VPS_1 и VPS_4 воркеры проигрывают или не дают
  преимущества: выше задержки, ниже фактический потолок RPS (~50 / ~190 против
  ~230–260 у main). На VPS_8 воркеры могут обогнать main по RPS, но с
  значительно худшей латентностью; при 500 RPS качество ломается (таймауты
  Telegram). Для лёгкого хендлера цена мощного железа и текущей схемы
  "DB-first + динамические Bot" на запрос **не окупается**.
- **Главная причина регресса AB-1** - это не воркеры сами по себе, а отказ от реестра
  в памяти для готовых объектов Bot в пользу обращения в БД и их создания на каждый
  запрос (см.2). Пока это не смягчено кэшем/шардированием, масштабирование
  числом процессов на одном сервере с БД слабо помогает.
- **AB-2, polling.** Корректность claim таблицы БД вместо реестра асинхронных мьютексов
  lock для сериализации чата подтверждена. Contention: ровно один успешный
  розыгрыш. По производительности ветки близки. Однако, сам режим поллинга при
  пределе 40 активных ботах (см.8) - тупиковый режим для роста сервиса.

## Рекомендации

- Не вводить текущий `feature/workers` в прод без доработки архитектуры.
  Триггер по нагрузке: к пику порядка ~100 RPS на тяжёлых сценариях (`/run`)
  должен быть готов уже **улучшенный** вариант, с повторным нагрузочным прогоном.
- Перевести мастер-бот на webhook с поллинга даже без полной ветки воркеров - это
  сразу снимает постоянную нагрузку на CPU.
- Вернуться к кэшу/реестру объектов Bot (упрощённое шардирование по воркерам, внешний
  кэш или иной вариант) и сократить DB-работу на каждый webhook до необходимости.
- Опционально: реализовать сценарий нагрузочного тестирования k6 без удержания
  частоты по факту получения обратного ответа (см.7). Это позволит оценить истинную
  устойчивость ветки main к нагрузке.
- Долгосрочно (не следует напрямую из метрик AB): рассмотреть стек с нормальной
  многопоточностью (например Go), если после кэша и разделения нагрузки сервиса
  и БД потолок стека Python/asyncio всё ещё недостаточен. Цена - переписывание проекта.
